# PRD: TeamMemory

## 1. Goal

TeamMemory는 프로젝트 문서, 회의록, 업무 메모, Todo, 이슈, 리스크, 보고서, 인수인계 정보를 프로젝트 단위로 모아 팀장이 빠르게 현재 상황을 파악하도록 돕는 운영 인텔리전스 서비스다.

단순한 "AI 분석 기능"이 아니라, 민감한 운영 데이터를 안전하게 수집하고, 누락된 데이터를 감지하며, AI 결과를 근거와 신뢰도 기반으로 검토할 수 있게 하는 것이 핵심이다.

## 2. Problem

현재 기획은 AI가 문서에서 무엇을 추출할 수 있는지에 초점이 있다. 하지만 실제 서비스 도입과 운영 성공을 위해서는 다음 리스크가 먼저 해결되어야 한다.

- 운영 데이터에는 내부 의사결정, 담당자, 병목, 리스크, 계약/고객 관련 정보가 포함될 수 있어 보안과 권한 설계가 선행되어야 한다.
- 파일 업로드 기반만으로는 팀원들이 꾸준히 데이터를 넣지 않을 가능성이 높다.
- AI가 추출한 Todo, 리스크, 요약을 근거 없이 보여주면 사용자가 믿지 않거나 과신할 수 있다.
- 데이터가 오래되었거나 중복되었거나 누락되면 AI 결과 품질이 빠르게 떨어진다.
- Dashboard만으로는 사용자가 문제를 제때 확인하지 못할 수 있으므로 알림과 에스컬레이션이 필요하다.

## 3. Users

- 팀장: 프로젝트 상태, 리스크, 지연 업무, 담당자별 병목을 확인한다.
- PM/운영 관리자: Todo, 이슈, 보고서, 데이터 수집률을 관리한다.
- 팀원: 문서, 회의록, 업무 메모, 이슈를 연결하거나 확인한다.
- 신규 입사자/인수인계 대상자: 권한이 허용된 범위 안에서 프로젝트 맥락과 히스토리를 확인한다.

## 4. Core Flows

1. 관리자가 팀, 프로젝트, 멤버, 권한 정책을 설정한다.
2. 시스템이 기존 업무 도구와 파일 업로드를 통해 프로젝트 데이터를 수집한다.
3. 수집된 문서는 민감정보 감지, 권한 범위 확인, 중복/오래된 문서 검사를 거친다.
4. 문서는 chunk로 분할되고, 프로젝트 및 권한 범위에 맞게 RAG 검색 인덱스에 연결된다.
5. AI가 Todo, 이슈, 리스크, 요약, 결정사항, 인수인계 항목을 추출한다.
6. 사용자는 AI 결과의 신뢰도, 근거 문장, 원본 문서 링크를 확인하고 승인/거절한다.
7. Dashboard는 진행률, 지연 Todo, High Risk, 데이터 수집률, 누락 영역을 보여준다.
8. 위험 또는 누락이 감지되면 Slack/Teams/메일 등으로 알림과 에스컬레이션을 보낸다.
9. 권한 기반 AI Assistant가 사용자의 권한 내 문서만 검색하여 답변한다.

## 5. Use Cases

### UC-00. 보안/권한/데이터 거버넌스

목적: 민감한 운영 데이터를 안전하게 수집, 분석, 보관, 삭제한다.

필수 기능:

- 사용자별 권한 관리: 팀장, PM, 운영 관리자, 팀원, 신규 입사자별 접근 범위를 제한한다.
- 프로젝트/문서별 접근 권한: 업로드한 사람만 볼 수 있는 문서와 조직 전체 공유 문서를 구분한다.
- 민감정보 마스킹: 주민번호, 전화번호, 이메일, 고객명, 계약금액 등 민감정보를 자동 감지하고 마스킹한다.
- AI 학습 사용 금지 옵션: 업로드 데이터가 외부 모델 학습에 쓰이지 않도록 설정과 약관에 명시한다.
- 감사 로그: 누가 어떤 문서를 업로드했고, 누가 어떤 질문을 했고, 어떤 답변을 봤는지 기록한다.
- 데이터 삭제 정책: 문서 원본, 분석 결과, chunk, vector DB 데이터를 완전 삭제할 수 있어야 한다.
- 배포 옵션: 온프레미스 또는 VPC 배포 옵션을 고려한다.
- 외부 LLM 사용 시 데이터 반출 범위와 저장 여부를 표시한다.

MVP 판단: UC-00은 편의 기능이 아니라 도입 전제 조건이다.

### UC-01. 데이터 수집/연동/업로드

목적: 팀원이 직접 업로드하지 않아도 기존 업무 도구에서 운영 데이터를 자동으로 모은다.

수집 방식:

- 파일 업로드: 회의록, 운영 문서, 메모, API 명세, 보고서.
- Slack/Teams 채팅 연동.
- 이메일 연동.
- Google Drive, SharePoint, Notion 문서 연동.
- Jira, Linear, Asana, Trello 이슈 연동.
- 회의록 자동 업로드.
- 정기 리마인드: "이번 주 운영 이슈를 등록해주세요."

핵심 요구사항:

- 문서별 source type을 저장한다.
- 수집 성공/실패 상태와 마지막 동기화 시간을 저장한다.
- 프로젝트별 수집 범위를 관리자가 제어한다.
- 연결된 외부 도구의 권한 범위를 사용자에게 명확히 보여준다.

### UC-02. Todo 관리

목적: 문서와 회의록에서 나온 액션 아이템을 공식 업무로 관리한다.

기능:

- AI가 Todo 후보를 추출한다.
- Todo 후보에는 신뢰도, 근거 문장, 원본 문서 링크를 표시한다.
- 사용자가 승인해야 공식 Todo로 반영된다.
- 담당자, 기한, 우선순위, 상태를 관리한다.
- 지연 또는 blocked 상태를 Dashboard와 알림에 연결한다.

### UC-03. 이슈/리스크 관리

목적: 프로젝트 운영 리스크와 장애 요인을 조기에 감지한다.

기능:

- AI가 문서, 회의록, 이슈 관리 도구에서 리스크 후보를 추출한다.
- High Risk, Blocked, 일정 지연 가능성, 반복 이슈를 분류한다.
- 각 리스크에 신뢰도, 근거 문장, 원본 링크를 표시한다.
- 담당자와 대응 상태를 관리한다.
- 같은 유형의 이슈가 반복되면 재발 리스크로 표시한다.

### UC-04. Dashboard

목적: 팀장이 프로젝트 상태를 한 화면에서 판단할 수 있게 한다.

기능:

- 프로젝트 진행률.
- 이번 주 Todo 전체/완료/지연 현황.
- 미해결 이슈와 High Risk.
- blocked 상태 지속 기간.
- 최근 보고서와 인수인계 점수.
- 데이터 수집률과 데이터 부족 경고.
- AI 결과 중 승인 대기 항목.

### UC-05. 보고서 생성

목적: 팀장/PM이 주간 보고와 상태 공유를 빠르게 만들 수 있게 한다.

기능:

- 주간 보고서 자동 초안 생성.
- Todo, 이슈, 리스크, 결정사항을 기반으로 요약한다.
- 근거 문서 링크를 포함한다.
- 데이터 부족 시 보고서 신뢰도 경고를 표시한다.
- 사람이 검토한 뒤 공식 보고서로 확정한다.

### UC-06. 지식 전달/인수인계

목적: 새 담당자나 신규 입사자가 프로젝트 맥락을 빠르게 이해한다.

기능:

- 프로젝트 히스토리, 주요 결정사항, 남은 Todo, 주요 리스크를 요약한다.
- 권한이 허용된 문서만 기반으로 생성한다.
- 오래된 문서를 기반으로 한 내용에는 경고를 표시한다.

MVP 판단: 데이터 품질과 권한 기반 RAG가 안정화된 뒤 효과가 커지므로 후순위로 둔다.

### UC-07. 권한 기반 AI Assistant

목적: 사용자가 자연어로 프로젝트 상태를 질문하고, 권한 내 문서를 기반으로 답변을 받는다.

필수 기능:

- 질문자의 권한에 따라 RAG 검색 범위를 제한한다.
- 답변 출처도 질문자가 접근 가능한 문서만 표시한다.
- 민감정보가 포함된 답변은 마스킹하거나 차단한다.
- 관리자용 질문/답변 감사 로그를 제공한다.
- 답변에는 근거 문서, chunk, 작성일, 신뢰도 또는 한계를 표시한다.

위험 예시:

- 신규 입사자가 "이번 달 조직 리스크 알려줘"라고 질문했을 때 권한 없는 문서까지 검색하면 안 된다.

### UC-08. 데이터 수집률 관리

목적: 팀원들이 데이터를 입력하지 않아 분석 품질이 떨어지는 문제를 감지한다.

기능:

- 팀/프로젝트별 최근 데이터 입력 현황을 표시한다.
- 7일 이상 업데이트 없는 영역을 표시한다.
- 누락된 회의록, 이슈, TODO를 감지한다.
- 담당자에게 자동 리마인드를 보낸다.
- 데이터 부족 시 AI Summary와 보고서에 신뢰도 경고를 표시한다.

예시 문구:

> 이번 주 PM 회의록이 없어 리스크 분석 신뢰도가 낮습니다.

### UC-09. 알림 및 에스컬레이션

목적: 사용자가 Dashboard를 보지 않아도 중요한 위험과 지연을 놓치지 않게 한다.

기능:

- High Risk 감지 시 PM/팀장에게 알림을 보낸다.
- blocked 상태가 N일 이상 지속되면 알림을 보낸다.
- 기한 지난 Todo를 담당자와 관리자에게 알린다.
- 같은 이슈가 반복 발생하면 재발 알림을 보낸다.
- 긴급 이슈는 Slack/Teams/메일로 발송한다.

## 6. Cross-Cutting Requirements

### AI 신뢰도와 근거

- AI 추출 결과별 신뢰도를 표시한다.
- 근거 문장과 원본 문서 링크를 제공한다.
- AI 결과는 기본적으로 "확정 전" 상태로 둔다.
- 사람이 승인해야 공식 Todo/이슈/보고서에 반영된다.

예시:

```text
결제 API 일정 지연 가능성 있음
신뢰도: 82%
근거: 5월 12일 회의록 중 "API 테스트가 다음 주로 밀릴 수 있음"
```

### 데이터 품질 관리

- 중복 문서를 감지한다.
- 오래된 문서를 표시한다.
- 문서 버전을 관리한다.
- 신뢰 가능한 소스 우선순위를 설정한다.
- 오래된 문서를 기반으로 한 답변에는 경고를 표시한다.

### 감사와 운영 관리

- 문서 업로드, 외부 도구 동기화, AI 질문, AI 답변 조회, 승인/거절 액션을 로그로 남긴다.
- 관리자는 프로젝트별 위험, 데이터 수집률, 승인 대기 AI 결과, 권한 위반 시도를 확인할 수 있어야 한다.

## 7. MVP Priority

| Priority | Use Case | Reason |
| --- | --- | --- |
| 0 | UC-00 보안/권한/데이터 정책 | 이거 없으면 도입 불가 |
| 1 | UC-01 데이터 수집/연동/업로드 | 데이터 없으면 서비스 무의미 |
| 2 | UC-08 데이터 수집률 관리 | 팀원이 안 넣는 문제 해결 |
| 3 | UC-02 Todo 관리 | 바로 업무 가치 있음 |
| 4 | UC-03 이슈/리스크 관리 | 운영 인텔리전스의 핵심 |
| 5 | UC-04 Dashboard | 집계 화면 |
| 6 | UC-09 알림/에스컬레이션 | 실제 운영에 필요 |
| 7 | UC-05 보고서 생성 | 팀장용 가치 |
| 8 | UC-07 AI Assistant | 권한/RAG 안정화 후 |
| 9 | UC-06 지식 전달 | 데이터가 쌓인 뒤 효과 있음 |

## 8. MVP Features

- 팀, 프로젝트, 멤버, 역할 관리.
- 프로젝트/문서 단위 권한 관리.
- 문서 업로드와 주요 업무 도구 연동을 고려한 source type 관리.
- 문서 chunk 저장과 vector store 참조 저장.
- AI 추출 Todo 후보와 승인/거절 workflow.
- 이슈/리스크 관리.
- 데이터 수집률 Dashboard.
- 기본 감사 로그.
- AI 결과의 신뢰도, 근거 문장, 원본 링크 표시.
- 주요 위험과 지연에 대한 알림.

## 9. Out Of Scope For First MVP

- 완전한 온프레미스 패키징.
- 모든 외부 업무 도구의 실시간 양방향 동기화.
- 고도화된 조직 전체 권한 정책 엔진.
- 자동 결재/승인 workflow.
- 고급 예측 모델.
- 조직 온보딩용 지식 전달 고도화.

## 10. Success Criteria

- 모든 핵심 업무 데이터가 `project_id` 기준으로 분리된다.
- 사용자의 권한 범위 밖 문서는 Dashboard, AI Assistant, 보고서, 출처 목록에 노출되지 않는다.
- AI 추출 Todo/이슈/리스크는 근거와 신뢰도 없이 공식 상태가 되지 않는다.
- 프로젝트별 데이터 수집률과 누락 영역을 확인할 수 있다.
- 데이터 부족 상태가 AI Summary와 보고서에 표시된다.
- 주요 High Risk, blocked Todo, 기한 초과 Todo에 대해 알림이 발생한다.
- 문서 원본, 분석 결과, chunk, vector DB 데이터 삭제 정책이 정의된다.

## 11. Team Lead Message

팀장님에게 전달할 핵심 문장은 다음과 같다.

> 파일 업로드 기반만으로는 데이터가 쌓이지 않을 가능성이 큽니다. 운영 인텔리전스 AI라면 기존 업무 도구에서 자동으로 데이터를 가져오고, 데이터 부족 상태를 감지하는 기능이 MVP에 포함돼야 합니다. 또한 운영 데이터는 민감도가 높기 때문에 보안/권한/감사 로그를 UC-00으로 먼저 정의해야 합니다.
