# OpsRadar2 2026-06-21 후속 인수인계

## 문서 범위

- 기준점은 2026-06-20의 `fa39e80 feat: AI 어시스턴트 출처 다운로드와 개인 대화 격리`이다.
- 이 문서는 `fa39e80..b3e1de3` 사이의 오늘 코드 변경과 오늘의 DB 정리/시딩/검증만 정리한다.
- 현재 브랜치에서는 기존 루트 `HANDOFF.md` 파일을 찾을 수 없었다. 따라서 이 문서는 이전 문서의 대체본이 아니라 위 기준점 이후의 독립적인 변경 기록이다.
- 자격증명, 비밀번호, DB 접속 정보는 포함하지 않는다.

  - `pre-handover-rework-20260621`

## 2. 오늘 코드 변경

### `53781e8` - AI Assistant 캘린더 그라운딩 고도화

- 파일: `app/services/operational_assistant_service.py`, `frontend/public/static/js/api-integration.js`, `frontend/public/static/js/handoff.js`.
- 수동 등록 여부와 무관하게 캘린더 일정을 Assistant 운영 근거로 포함한다.
- 일정은 최근 90일과 향후 180일을 조회하고, 질문의 제목/담당자 키워드로 랭킹한다.
- 기간 부재처럼 여러 행으로 저장된 일정은 시작일~종료일 한 건으로 그룹화해 답변한다.
- fallback 답변에도 `일정 근거`를 넣어 AI 모델이 연결되지 않은 경우에도 일정 제목, 기간, 담당자, 유형을 보여 준다.
- 종일 일정은 `00:00`, `00:00:00`, 빈 값, `--:--`을 시간 미지정으로 처리한다. 캘린더 제목에는 실제 유효 시간(`HH:MM`, 단 `00:00` 제외)만 표시한다.
- 검증: 라이브 DB에서 Assistant 일정 조회 SQL 실행 성공. ZZTEMP 문서/청크는 각각 0건으로 확인했다.

### `8f5ac78` - 캐시버스터 갱신

- 파일: `frontend/public/index.html`.
- `handoff.js`와 `api-integration.js` 정적 캐시버스터를 갱신했다.
- 정적 JS/CSS 변경 뒤에는 `index.html`의 `?v=`도 함께 올려 브라우저 캐시로 인한 구버전 실행을 막는다.

### `199fc6c` - 담당자 자동배정 하드코딩 폴백 제거

- 파일: `app.js`, `todo-calendar-enhancements.js`, `ops-workflow-enhancements.js`, `api-integration.js`.
- 근거가 없을 때 특정 사용자 이름이나 `members[0]`을 자동 담당자로 넣던 폴백을 제거했다.
- `recommendRiskAssignee()`의 최종 폴백은 빈 문자열이며, 근거가 없으면 UI는 `미지정`으로 남는다.
- 담당자 확정은 사용자 선택 또는 명시적인 문서/업무 근거가 있을 때만 한다.

### `b3e1de3` - 캘린더/운영로그 화면 UI 정리

- 파일: `frontend/src/react-mount/CalendarScreen.jsx`, `frontend/src/react-mount/AnalysisScreen.jsx`, `frontend/public/index.html`, `frontend/public/static/js/app.js`.
- 캘린더 오른쪽 패널의 리스크/부재/마일스톤 영역을 제거하고 일정 등록 UI를 하단 배치했다.
- 운영로그 분석 결과의 `Todo 확인`, `이슈 확인` 버튼을 제거했다.
- 매우 중요: 캘린더와 운영로그 분석 화면의 실제 소스는 React JSX다. `index.html`의 동일 영역을 고쳐도 React 렌더링이 덮어써서 효과가 없다.
- JSX를 수정한 뒤 반드시 `frontend`에서 `npm run vite:build`를 실행한다.
- 생성되는 `public/static/react/main.js`는 Git ignore 대상이다. 체크아웃/clone 후에는 항상 재빌드해야 React 화면에 변경이 반영된다.

## 3. 인수인계센터 현황

- 현재 적용된 것은 vanilla 모드의 `frontend/public/static/js/handoff.js`다.
- `window.__HANDOFF_REACT_ENABLED__`가 없으면 하드코딩 `DEMO_SOURCE`, 데모 카드, localStorage 기반 화면이 동작한다.
- LLM 호출, 실제 Todo/Issue API 연결, 서버 저장/공유는 아직 없다.
- 실기능화 순서:
  1. 하드코딩 데이터 소스를 Todo/Issue/문서 API 데이터로 교체
  2. `buildPreviewData()` 성격의 데모 조립을 실제 LLM/서버 생성 API로 교체
  3. 공유 toast를 서버 저장 및 수신자 전달 기능으로 교체
  4. 필요 시 `frontend-handoff-dev-integration`의 React 세트를 파일 단위로 검토해 통합
- 전체 브랜치 병합은 금지한다. 현재는 vanilla 데모만 도입된 상태다.

## 4. AI Assistant와 보고서 검증

- Assistant는 근거가 없는 질문에 데이터 없음/확인 필요로 답해 환각을 피하는 동작을 확인했다.
- 문서 검색은 패러프레이즈 질문도 통과했으며 현재 유사도 임계값 `0.41`은 유지한다.
- 수동 캘린더 일정의 제목/담당자 질문은 위 일정 랭킹 로직으로 답변 근거에 포함된다.
- 월간보고서는 2026년 4월, 5월, 6월에 대해 생성 가능하며 출처 문서 연결과 사실 기반 제약이 동작하는 것을 확인했다.
- 알려진 약점:
  - 보고서 작성자 표시가 `WorkRadar`로 고정된다. 요청자 id를 보고서 생성 경로에 전달하지 않는다.
  - `summarizer.py`의 description 500자 제한은 의도된 설계다.
  - 보고서 생성은 주간/월간 고정이며 임의 기간 선택은 지원하지 않는다.

## 6. 구조적 메모와 다음 작업

## 7. 작업 시 주의사항

- DB 스키마는 사용자 지시 없이 변경하지 않는다.
- API 호출은 프런트 숫자 UI id가 아닌 UUID `apiId`를 사용한다.
- `.env`, 비밀번호, API 키, DB 접속 정보, 백업, 임시 스크립트는 커밋하지 않는다.
- `workflow-v2.js` 수정 후에는 `node --check frontend/public/static/js/workflow-v2.js`를 실행한다.
- 현재 작업 트리의 `frontend/package.json` 수정 표시는 줄바꿈(EOL) 상태다. 내용 diff가 없으므로 스테이징하지 않는다.
