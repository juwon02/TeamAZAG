## 2. 이번 변경 요약

### 신입 온보딩: 실제 업무 적응 플랜으로 전환

- 신규 React 컴포넌트: `frontend/src/react-mount/handover/OnboardingLearningPlan.jsx`.
- 온보딩 대상은 사용자 선택 대상이 아니라 항상 `신입`으로 고정한다. 사수와 소속 팀, 참고 기간만 선택한다.
- 온보딩 후보 자료는 하드코딩 고객사/품목 목록 대신 API에서 받은 **진행 Todo와 미해결 Issue**다. API 실패 시에만 기존 fallback 후보를 쓴다.
- 온보딩 초안은 `3 Step으로 배우는 업무 적응 플랜 - 신입`으로 표시한다.
  - `1 Step`: 흐름과 자료 이해
  - `2 Step`: 사수와 실제 업무 연습
  - `3 Step`: 독립 처리 범위와 위험 기준 합의
- 실전 연습 Todo와 사수 확인 항목은 클릭 영역 전체가 아닌 별도 진행 버튼으로 체크한다. 진행률은 브라우저 localStorage에만 저장되며 업무 상태/DB Todo를 바꾸지 않는다.
- 체크리스트 생성 시 `참고 자료`, `사수 확인 질문`, 문서 파일명, Markdown 기호(`'''`, `````, `**`) 같은 구조 표식은 제거한다.
- 사수 영역 제목은 `사수 확인 및 참고 자료`다. 미해결 Issue는 `사수 검토가 필요한 리스크`로 별도 표기한다.
- 온보딩 저장본도 미리보기와 동일한 `OnboardingLearningPlan` 컴포넌트로 렌더링한다. 저장 후 화면의 가독성이 미리보기보다 떨어지던 문제를 피한다.

### 온보딩 AI 생성 경로

- `POST /api/v1/knowledge/generate-handover`는 기존 인수인계 API를 재사용한다.
- 요청에 `target`, `mentor`를 추가하고 `type=onboarding`이면 `HandoverDraftService`의 `ONBOARDING_PROMPT`를 사용한다.
- 백엔드 입력은 기존 `gather_handover_input()`의 선택된 Todo/Issue/출처 문서 기반이다. 새 영속 모델이나 스키마는 추가하지 않았다.
- 온보딩 LLM 결과는 5개 섹션을 모두 포함해야 AI 결과로 채택한다. AI가 실패하거나 Azure가 비활성화면 선택된 실제 자료 기반의 frontend fallback 플랜을 표시한다.
- AI 결과라도 원문에 없는 고객사, 품번, Todo, 담당자, 완료 기준을 만들지 않도록 프롬프트를 제약한다. 근거가 부족하면 `확인 필요` 또는 `사수와 범위 합의 필요`를 사용한다.

### 업무 인수인계 화면 정리

- `HandoffPreview.jsx`를 운영 문서형 레이아웃으로 바꿨다. 개요, 우선순위/즉시 조치, 진행 업무/이슈, 참고 자료를 읽기 쉬운 블록으로 구분한다.
- 저장 문서도 동일한 레이아웃을 사용한다. 기존의 저장 화면 전용 카드 그리드는 제거했다.
- 편집 모드는 섹션별 textarea를 유지한다. 저장 문서 삭제/복제는 localStorage 문서함에만 적용된다.
- `documentDownload.js`를 추가해 React 인수인계/온보딩과 legacy 화면이 인증 토큰 저장 형식(`access_token`, `token`, 세션 객체)을 모두 처리하며 출처 파일을 blob 다운로드한다.
- `workflow-v2.js`의 legacy 출처 다운로드도 중앙 `opsRadarApi.downloadDocument()`를 우선 사용하고, 없을 때 Authorization 헤더를 붙이는 fallback을 둔다.

## 3. Todo 담당자/마감일 추천의 근거 강화

### 분석 파이프라인

- `app/ai/summarizer.py`:
  - LLM에게 담당자는 원문에 명시된 실명일 때만, 마감일은 원문에 명시된 날짜/상대 기한일 때만 반환하도록 지시한다.
  - `_grounded_assignee()`는 반환 이름이 실제 원문에 있을 때만 허용한다.
  - `_grounded_due_date()`는 `YYYY-MM-DD`, `M/D`, `M월 D일`, 오늘/내일/N일 내·후 근거를 검증한다. 날짜 근거가 없으면 `None`이다.
- `app/services/document_service.py`:
  - 활성 프로젝트 멤버와 원문 실명이 동시에 일치할 때만 `assignee_member_id`를 저장한다.
  - 검증된 `YYYY-MM-DD`만 `due_at`으로 저장한다.

### 프론트 추천/입력 UX

- `api-integration.js`, `ops-workflow-enhancements.js`의 역할/키워드 기반 개인 자동배정을 제거했다. 원문에 실제 이름이 없으면 담당자는 `미지정`이다.
- `todo-calendar-enhancements.js`는 기존 마감일 또는 제목/내용의 명시 날짜만 추천한다. 날짜 근거가 없으면 오늘 날짜를 입력하고 해당 날짜 입력란에 포커스하며, 사용자에게 근거 없음 안내를 표시한다.
- 이슈 대응 Todo 생성도 위험도에 따른 임의 `+7일` 대신 같은 원칙을 따른다.
- 로컬 날짜를 직접 조립해 UTC 변환으로 날짜가 하루 밀릴 가능성을 줄였다.

## 4. Todo 설명 잘림 진단 - 미수정, 다음 작업 최우선

### 확인 결과

- 라이브 DB의 시딩 Todo 13개를 SELECT로 조회했다. 설명 길이는 109~137자, 490~510자 항목은 0개, DB에서 실제 `...`로 끝나는 항목은 0개였다.
- `todos.description`은 PostgreSQL `text`이며 컬럼 길이 제한은 원인이 아니다.
- 현재 13개 모두 72자 초과라 브라우저 축약 규칙의 영향을 받는다.

### 실제 원인과 위험

- `frontend/public/static/js/api-integration.js`의 `briefTodoApiDescription()`이 API 응답을 72자로 자르고 `todo.description` 자체에 넣는다.
- `frontend/public/static/js/app.js`의 `openEditModal()`도 수정 textarea에 `briefTodoText()`(72자 + `...`)를 넣는다.
- 수정 모달을 열어 담당자만 바꾸고 저장해도 `api-integration.js`/`todo-calendar-enhancements.js`의 PATCH가 축약본을 DB에 덮어쓸 수 있다. 현재 문제의 핵심은 표시 축약이 저장 손실로 이어지는 구조다.
- `summarizer.py`의 `_detailed_todo_description()`와 `_with_subject_description()`에는 별도의 500자 상한도 있다. 현재 시딩에는 영향이 없지만 긴 신규 원문은 저장 시 문장 중간에서 잘릴 수 있다.
- Issue의 `risk_reason`은 별도 Text 컬럼이며 문서 분석 자동 저장은 이 값을 채우지 않는다. Issue 설명도 `_with_subject_description()`을 쓰므로 500자 상한 대상이다.

### 권장 수정 순서

1. `normalizeTodo()`는 원문 `description`을 보존하고, 카드 전용 `descriptionPreview`를 별도 필드로 만든다.
2. 수정/상세 모달과 PATCH는 항상 원문 description을 사용한다. 목록만 CSS line-clamp 또는 preview 필드로 축약한다.
3. 500자 제한은 제거하거나 충분한 상한 및 문장 경계 절단으로 바꾼다.
4. 과거에 수정 저장된 Todo는 원문이 유실되었을 수 있다. 일괄 UPDATE하지 말고 연결 문서/청크로 개별 복구 여부를 검토한다.

## 5. 검증 및 작업 규칙

- 커밋 전 실행할 검증:
  - `python -m py_compile app/ai/summarizer.py app/api/v1/endpoints/knowledge.py app/services/document_service.py app/services/handover_draft_service.py`
  - `node --check frontend/public/static/js/api-integration.js`
  - `node --check frontend/public/static/js/ops-workflow-enhancements.js`
  - `node --check frontend/public/static/js/todo-calendar-enhancements.js`
  - `node --check frontend/public/static/js/workflow-v2.js`
  - `cd frontend && npm run vite:build`
- 정적 JS/CSS를 수정하면 `frontend/public/index.html`의 캐시버스터도 함께 올린다.
- `workflow-v2.js`는 IIFE override 파일이므로 문법 오류 하나로 전체 override가 미등록될 수 있다. 수정 후 `node --check`가 필수다.
- DB 스키마 변경 금지. API 호출에는 화면 숫자 id가 아닌 UUID `apiId`를 사용한다.
- `.env`, `.env.local`, API 키/비밀번호/DB 접속 정보, `backups/`, `dummy_data/`, 임시 검사 스크립트(`check_pw.py`, `check_names.py`)는 커밋 금지다.
- `frontend/package.json`의 변경 표시는 EOL만의 변경으로 확인되었으므로 이번 커밋에서 제외한다.
